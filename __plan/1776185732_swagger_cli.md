# Refined Plan: Swagger CLI Integration

## Objectives
Automate the generation of API documentation using the NestJS Swagger CLI Plugin. This will reduce manual boilerplate in Controllers and DTOs while ensuring consistent, high-quality documentation with automatic payload and response examples.

## Prerequisites
- **Node.js**: 22+ (Project standard)
- **Package Manager**: pnpm (Project standard)
- **Dependencies**: `@nestjs/swagger`, `nestjs-zod`, and `@scalar/nestjs-api-reference` are already installed and used in `src/main.ts`.

## Detailed Flow

### 1. Enable NestJS Swagger CLI Plugin
- **Task**: Update `nest-cli.json` to enable the `@nestjs/swagger` compiler plugin.
- **Configuration**:
  ```json
  "compilerOptions": {
    "deleteOutDir": true,
    "plugins": [
      {
        "name": "@nestjs/swagger",
        "options": {
          "classValidatorShim": false,
          "introspectComments": true,
          "dtoFileNameSuffix": [".dto.ts"],
          "controllerFileNameSuffix": [".controller.ts"]
        }
      }
    ]
  }
  ```
- **Action**: Add the `plugins` array to the `compilerOptions` section in `nest-cli.json`.
- **Validation**: Run `pnpm run build` and ensure no compilation errors occur.

### 2. Patch Swagger for Zod Support
- **Task**: Enhance `src/main.ts` to better integrate `nestjs-zod` with the Swagger metadata.
- **Implementation**:
  - Import `patchNestjsSwagger` from `nestjs-zod`.
  - Call `patchNestjsSwagger()` at the top of the `bootstrap` function.
- **Note**: While `cleanupOpenApiDoc` is already used, `patchNestjsSwagger` ensures that the CLI plugin's automated metadata correctly maps to Zod DTOs.

### 3. Enrich DTO Metadata
- **Task**: Ensure Zod schemas in `src/**/*.dto.ts` provide sufficient metadata for the documentation.
- **Best Practices**:
  - Use `.describe('...')` on Zod properties for clear descriptions.
  - The CLI plugin will automatically handle the mapping of TypeScript types to OpenAPI types, but Zod schemas remain the "source of truth" for validation and detailed metadata.
- **Validation**: Check the "Model" section in Scalar UI (`/api`) to verify that field descriptions and types are correctly displayed.

### 4. Optimize Controllers
- **Task**: Simplify `src/**/*.controller.ts` by removing redundant Swagger decorators.
- **Simplification**:
  - Leverage `introspectComments: true` to use JSDoc comments (`/** ... */`) as endpoint summaries/descriptions instead of manual `@ApiOperation()`.
  - Rely on the CLI plugin to infer `@ApiResponse()` from explicit return types (e.g., `async create(...): Promise<CreatePartnerDto>`).
- **Validation**: Ensure that endpoints still have summaries, descriptions, and correct status codes in the Scalar UI.

### 5. Verification & Testing
- **Task**: Perform a final audit of the generated API documentation.
- **Checklist**:
  - [ ] Access Scalar UI at `/api`.
  - [ ] Verify that "Request Body" and "Responses" sections have interactive examples.
  - [ ] Ensure all endpoints are correctly tagged based on their Controller names.
  - [ ] Confirm that Bearer Auth (`addBearerAuth`) is correctly displayed for protected routes.

## Technical Notes
- **Introspection**: JSDoc comments on controller methods will be automatically picked up as `summary` (first line) and `description` (remaining lines).
- **Zod Integration**: Since the project uses `nestjs-zod`, the CLI plugin's primary benefit is for Controller automation and ensuring DTO types are correctly registered in the OpenAPI `Components` section.
- **Build Process**: The plugin works at compile-time, so ensure `dist/` is cleaned and rebuilt during testing.
