# Refined Plan: Partner DTO Implementation

## Objectives
Create a comprehensive set of Data Transfer Objects (DTOs) for the Partner resource using `nestjs-zod`. These DTOs will serve as the standardized response formats for the Partner service and repository, ensuring type safety and consistent API documentation.

## Prerequisites
- Knowledge of `src/partner/partner.repository.ts` implementation (specifically `findAll` and `findById`).
- Understanding of `Partner`, `User`, `PartnerContact`, `PartnerBankAccount`, and `Bank` models in `prisma/schema.prisma`.
- Established patterns in `src/_core/types/response.type.ts`.

## Detailed Flow

### 1. Define Base Component Schemas
In a new file `src/partner/dto/partner.dto.ts`, define the following Zod schemas and their inferred types:

- **`PartnerUserSchema`**: Represents the user summary included in partner responses.
  - Fields: `id`, `username`, `email`, `phone_number` (optional), `role`.
- **`PartnerContactSchema`**: Represents `PartnerContact` model.
  - Fields: `id`, `name`, `email`, `phone_number`, `created_at`, `updated_at`.
- **`PartnerBankAccountSchema`**: Represents `PartnerBankAccount` model including nested `Bank`.
  - Fields: `id`, `bank_id`, `account_number`, `account_name`, `created_at`, `updated_at`.
  - Nested: `bank` (using a subset of `Bank` fields: `id`, `code`, `name`).
- **`PartnerBaseSchema`**: Core partner fields common to most responses.
  - Fields: `id`, `name`, `types` (array of `PartnerType`), `legal_entity`, `company_email`, `company_phone`, `created_at`, `updated_at`.

### 2. Define Compositional Schemas & DTOs
Using the base schemas, create the following DTO classes:

- **`PartnerDto`**: Full partner details without user association.
  - Includes: `PartnerBaseSchema`, `PartnerAddressSchema` (from `partner-create-payload.dto.ts`), `contacts` (array), `partner_bank_accounts` (array).
- **`PartnerFullDto`**: The "master" DTO including all relations.
  - Includes: `PartnerDto` fields + `user` (using `PartnerUserSchema`).
- **`PartnerResponseDto`**: Simplified DTO for non-admin `findAll` results.
  - Match `PartnerRepository.findAll` non-admin select: `id`, `name`, `types`, `legal_entity`, `company_email`, `company_phone`, `created_at`, `updated_at`.
- **`PartnerAdminResponseDto`**: Comprehensive DTO for admin `findAll` results.
  - Match `PartnerRepository.findAll` admin select: Full `PartnerBaseSchema`, `PartnerAddressSchema`, `deleted_at`, and `user` summary (id, username, email).

### 3. Implementation Guidelines
- Use `z.infer<typeof Schema>` for internal TypeScript typing.
- Use `export class XxxDto extends createZodDto(XxxSchema) {}` for NestJS/Swagger compatibility.
- Ensure all fields have `.describe()` calls for clear OpenAPI documentation.
- All date fields (`created_at`, `updated_at`, `deleted_at`) should use `z.date()`.

## Technical Notes
- **File Location**: All new DTOs should reside in `src/partner/dto/partner.dto.ts`.
- **Schema Re-use**: Reuse `PartnerAddressSchema` from `src/partner/dto/partner-create-payload.dto.ts` to maintain consistency in address field definitions.
- **Strictness**: Use `.strip()` (default Zod behavior) to ensure no extra fields are leaked, but consider `.strict()` for internal validation if needed.

## Verification Steps
1. **Schema Mapping**: Cross-reference `PartnerAdminResponseDto` with `PartnerRepository.findAll`'s `isAdmin` select block to ensure 1:1 field parity.
2. **Type Compatibility**: Ensure `PartnerFullDto` is compatible with the return type of `PartnerRepository.findById(id, true)`.
3. **Swagger Check**: (Manual) Once applied, verify that the DTOs appear correctly in the Swagger UI with the provided descriptions.
